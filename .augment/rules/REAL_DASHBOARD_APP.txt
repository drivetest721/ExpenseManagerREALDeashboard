Chaining:
 - This is the most critical task and heart of the application lies in this workflow.
 - An employee can apply for new Reimbursement through the page. then this application notified to thier 1st manager. and then that manager eithe approve or soft reject.
 - After Approve: 
     > the application approved by 1st manager of Base-Employee/Initiator then this applocation goes to 2nd manager of Initiator (manager of manager of Initiator) this chain goes until the Owner becase the nth manager of n-1 th manager of every employee is Owner it-self. If the Onwer Approve the Application then it will go to CA by notification system. The CA either Paid the Employee or reject the Application.
 - Soft Reject
     == The term Soft Reject is Highly dangerous so we use query or Ask
   Type 1 : Query
     > This Word means the Manger return the application to the Initiator and query (ask some question or leads the missing data. Example the Initiator not upload the iage correctly then manager reject/query that "You have not uploaded the Document properly please Upload it properly."),
     > This action do not notify it next manager. It Notify The Initiator and then  Initiator Can Re Apply after updating Details. and this notify the Manager who is Currently Query the Reimbursement.
     > To Query the Reimbursement employee required valid Description.

   Type 2: Ask
     > This action is Same as Query but it's Private. It Exactly Notify Initiator and Do not notify the Next manger.
 - Note: 
     > If the xth Manager is part of Reimbursement and query the In Initiator then it's only Notified to teh Initiator only and Re apply by Initiator can only Notify to the xth Employee not (x-1)th Emplyee or (x-2)th Employee whcih is part of Reimbursement cycle.

Status System:
  Status	Meaning
  DRAFT:	Saved In DB but do not forward it to manager or anyone it' visible in the draft
  SUBMITTED:	Sent to first manager of current User.
  IN_REVIEW:	Current manager reviewing
  QUERY_RAISED:	Public query
  PRIVATE_ASK:	Private query
  REAPPLIED:	User resubmitted. this notification goes to the Person who query the Reimbursement
  OWNER_APPROVED:	Approved by owner
  CA_PENDING:	Waiting for accountant
  CA_QUERY:	CA asked clarification
  CA_REAPPLIED	User replied to CA
  PAID:	CA marked paid
  PAYMENT_ACKNOWLEDGED:	Employee confirmed payment
  REJECTED	Permanently rejected
  AUTO_REJECTED:	SLA timeout
  CLOSED:	Final state

State Transition Rules
  Current State	| Action	|Next State
  DRAFT	|Submit	|SUBMITTED
  SUBMITTED	| Approve	| NEXT_MANAGER_PENDING
  SUBMITTED	|Query	| QUERY_RAISED
  QUERY_RAISED	|Reapply	| IN_REVIEW
  OWNER_APPROVED	|Send to CA	| CA_PENDING
  CA_PENDING	|Paid	| PAID
  PAID	|Acknowledge	|CLOSED

Workflow Rules
  Approval Flow
  Submit
    Employee submits reimbursement.
    System identifies manager hierarchy dynamically.
    First manager becomes current reviewer.
    Only current reviewer can approve/query/ask.

  Approve
    Move reimbursement to next manager.
    If next reviewer is owner:
    after approval move to CA.
    If no next reviewer:
    move directly to CA.

  Query
    Public visible message.
    Requires description.
    Sent back ONLY to initiator.
    Re-apply returns ONLY to querying manager.

  Ask
    Private communication.
    Visible only to:
    sender
    initiator
    owner/admin

  CA Query
    Allowed only one time.
    After initiator re-applies:
    CA only has:
    Paid
    Reject

  Payment
    CA marks reimbursement as PAID.
    Employee must acknowledge.
    After acknowledgment:
    reimbursement CLOSED.

  SLA Rules
    Configurable
    {
      "approval_sla_days": 3,
      "query_response_days": 2
    }

SLA Engine Rules

  Detect overdue reimbursements.
  Send reminder notification.
  Send escalation email to admin.
  Auto reject query if employee fails to respond in configured days.
